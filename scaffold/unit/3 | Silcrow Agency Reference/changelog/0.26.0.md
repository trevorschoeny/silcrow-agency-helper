# 0.26.0

**Date:** 2026-05-04

## What changed

- The plugin now ships a **changelog**: `3 | Silcrow Agency Reference/changelog/0.X.Y.md` — one file per version, describing what changed and any migration notes. New and existing agencies get the full backfilled history (0.20.0 onward).
- New file: `3 | Silcrow Agency Reference/Plugin Version.md` — a simple markdown record of which silcrow version the agency is currently synced with. Updated by `:silcrow-update` after a successful audit.
- `scaffold.sh` reads the current plugin version from `plugin.json` and substitutes `{plugin_version}` into `Plugin Version.md` when scaffolding a new agency.
- Registrar Update Workflow gains a final step (Step 10): update `Plugin Version.md` to reflect the version the agency just synced to, so future audits can quickly see "where we are."
- Audit Checklist A gains a check that `Plugin Version.md` exists and contains a parseable version.

## Migration notes

Existing agencies don't have `Plugin Version.md` or the `changelog/` folder yet. `:silcrow-update` proposes both as additions on next sync. After approval, the file reflects the agency's current synced version.

The change does **not** make `:silcrow-update`'s sync flow version-aware. The flow is still dynamic-diff (compare plugin source to agency state directly) per §0013. The version tracker and changelog are informational — they help humans and the Registrar quickly see "where we are" and "what changed" without changing the diff mechanics. If the sync flow is ever refactored to USE the changelog (enumerate changes by version rather than diff), that's a §0013 supersession at that point.
