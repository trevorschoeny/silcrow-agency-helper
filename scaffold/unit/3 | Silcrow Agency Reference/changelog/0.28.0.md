# 0.28.0

**Date:** 2026-05-04

## What changed

- **Removed `version` field from `plugin.json`.** silcrow now uses commit-SHA-based versioning. Each commit becomes a new "version" from Claude Code's perspective, which means the plugin cache invalidates on every commit automatically — no more manual `/plugin marketplace update` + nuke-and-reinstall ritual to propagate changes.
- **`scaffold.sh` reads the plugin's git commit SHA** (via `git -C "$PLUGIN_ROOT" rev-parse --short HEAD`) instead of parsing a version field. Substitutes the SHA into `Plugin Version.md` when scaffolding.
- **`Plugin Version.md` reformatted to track SHAs.** New format:
  ```
  **Currently synced with:** silcrow at commit `<sha>`
  **Sync date:** YYYY-MM-DD
  ```
  The body explains the SHA-based model and points readers at the `changelog/` for human-readable summaries.
- **Changelog filenames stay semver-style** (`0.28.0.md`, etc.) — but as **labels**, not pins. The `changelog/README.md` now explains: these names are bookkeeping aids for humans navigating the change history, not version pins anchored anywhere in the plugin. The git SHA is the canonical identifier.
- **Registrar Update Workflow Step 10** updated to read the SHA after a sync and write it to `Plugin Version.md` (separate commit from the audit ADR commit).
- **Audit Checklist A** check for `Plugin Version.md` updated to look for the new SHA-based format. Agencies still on the old `silcrow X.Y.Z` format are flagged for migration.

## Why

Three documented Claude Code bugs prevented the plugin cache from refreshing reliably when `plugin.json` had an explicit version field — even after `/plugin marketplace update`:

- [#16866](https://github.com/anthropics/claude-code/issues/16866) — `/plugin marketplace update` doesn't actually re-fetch the plugin cache.
- [#17361](https://github.com/anthropics/claude-code/issues/17361) — `installed_plugins.json` doesn't update even with `autoUpdate: true`.
- [#14061](https://github.com/anthropics/claude-code/issues/14061) — `/plugin update` doesn't invalidate the cache directory.

When `plugin.json` has no `version` field, Claude Code uses the git commit SHA as the cache key. Each commit produces a unique key, and stale entries are skipped. The cache invalidation problem disappears.

## Migration notes

- **Existing agencies** will have `Plugin Version.md` in the old format (`silcrow 0.27.0` or earlier). After running `:silcrow-update` against this version, the Registrar will update the file to the new SHA-based format as part of Step 10.
- **The plugin's marketplace entry** (`marketplace.json` in your `claude-plugins` repo, if you have one with version pins for silcrow) should also be checked — if it has explicit version refs, those may keep the cache pinned. Removing them aligns with the SHA-based model.
- **Existing changelog entries** (`0.20.0.md` through `0.27.0.md`) are unchanged. They're labels for the change sets that shipped under those names; subsequent entries continue the convention.
