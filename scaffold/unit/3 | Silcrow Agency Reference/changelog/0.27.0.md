# 0.27.0

**Date:** 2026-05-04

## What changed

- `:silcrow-update` and `:silcrow-add-unit` now accept invocation from either the unit's root directory **or** any agent directory inside the unit (Lead, Implementer, Registrar, User). Previously they required the unit root, which meant a Registrar or Lead invoking from their own session got bounced.
- Resolution logic: if CWD's basename starts with `@`, CWD itself is the unit. If CWD's basename matches `<X> @ <Y>` (an agent directory), the unit is `<cwd>/..` — one deterministic step, not iterative walking. Per §0012's flat layout, every agent directory lives directly inside its unit.
- This matches the pattern `:silcrow-add-agent` already used (it accepts the Lead's directory and uses `..` as the unit). Now all four skills handle agent-session invocation consistently.
- The "no walking up" principle is unchanged: it was always specifically about iterative *search* through ancestors. A single `..` step that resolves the unit by structural definition (agent dirs always live in their unit) is direct addressing, not search.

## Migration notes

None — this is a permissiveness extension, not a behavior change. Existing invocations from the unit root still work exactly as before. Existing invocations from agent directories now work where they previously got refused.
