# 0.27.0

**Date:** 2026-05-04

## What changed

- **`:silcrow-update` is now strictly root-Registrar-only.** Previously the skill accepted CWD as either the unit root OR any agent directory inside a unit. That flexibility was wrong — it allowed any agent (Lead, Implementer, User, custom agents, sub-unit Registrars) to invoke. The skill now refuses unless CWD is the agency's root unit Registrar directory (`@ <Agency Name>/Registrar @ <Agency Name>/`), with explicit refusal messages tailored to each failure mode.
- **`:silcrow-add-unit` is now strictly Lead-only.** Same shape of fix: CWD must be the parent unit's Lead directory. Adding a sub-unit is a substantive architectural decision (§0008 roster change); §0009 prohibits non-Lead roles from authoring it. Mirrors the pattern already in place for `:silcrow-add-agent` since 0.22.0.
- **Why the lockdown.** Both skills had failure modes when invoked from natural sessions (a Lead in `Lead @ Pebble/`, a Registrar in `Registrar @ Pebble/`) — they'd error with confusing messages about "CWD isn't a unit." The fix isn't to add MORE flexibility (accepting any role's session) — it's to make the role expectation explicit per skill. Each substantive skill names exactly which role should invoke and refuses otherwise.
- **`:silcrow-update`'s scope is now agency-wide.** Since only the root Registrar runs it, the audit scope is the entire agency tree (root + every sub-unit at every depth) — not just the subtree of the invoking unit. The Registrar Update Workflow doc reflects this: the audit-as-agency-wide-operation is a deliberate departure from §0012's standard federation rule, justified because the plugin defines agency-wide canonical state.
- **`:silcrow-update`'s message destination changed.** The Registrar (running the skill in their own session) now drops the trigger message into their own inbox at `<cwd>/inbox/`, archives it per §0005, and proceeds to the 10-step workflow in the same session — rather than dropping into a separate Registrar dir from a unit-root session.

## Migration notes

- **Existing agencies** can use `:silcrow-update` immediately under the new rule: open a session inside `@ <Agency Name>/Registrar @ <Agency Name>/` and invoke. Same for `:silcrow-add-unit` from the parent unit's Lead session (e.g., `Lead @ <Parent Unit>/`).
- **Old skill behavior is gone.** Sessions opened from the unit root (basename starts with `@`) will now refuse for both skills. This is intentional — the role-specific session is the canonical invocation point.
- **Sub-unit Registrars never run `:silcrow-update`.** The skill refuses if CWD's parent unit is a sub-unit (not the agency root). Plugin updates are agency-wide; sub-unit Registrars handle local audits only.
