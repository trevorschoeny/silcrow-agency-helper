# 0.29.0

**Date:** 2026-05-04

## What changed

- **Reverted 0.28.0's commit-SHA-based versioning.** The `version` field is back in `plugin.json`. `scaffold.sh` reads from it again. `Plugin Version.md` is back to the `silcrow X.Y.Z` format. The `changelog/0.28.0.md` entry is removed (the change set never lived as a stable release).

## Why

0.28.0 removed the `version` field with the goal of fixing a cache invalidation problem that prevented updates from propagating to existing agencies. After shipping it, the cache problem persisted anyway — turns out the propagation issue lives at a deeper layer than version-vs-SHA resolution. Specifically:

- The Claude Code desktop app pins plugin state at session startup, regardless of how the version is resolved.
- Even with SHAs producing unique cache keys per commit, an already-running session keeps using whatever it loaded at startup.
- The only reliable way to pick up updates is to fully quit and reopen the desktop app — and that's true whether the plugin uses version-based or SHA-based versioning.

So 0.28.0 traded a cleaner mental model (named releases that humans can navigate) for an uglier one (SHAs in `Plugin Version.md`) without actually solving the underlying problem. Reverting puts the named-release mental model back.

## What we learned

- The cache propagation problem is at the desktop app's session-state layer, not the plugin's version-resolution layer.
- Workaround for now: `cmd+Q` and reopen the desktop app between updates. This is necessary regardless of versioning model.
- Going forward: stay on version-based releases (the model in 0.27.0 and earlier). Bump the version intentionally with each meaningful change set; that's the established workflow.

## Migration notes

- **Existing agencies on 0.28.0** (the SHA-based release) will see their `Plugin Version.md` migrate back to the `silcrow X.Y.Z` format the next time `:silcrow-update` runs. The Registrar handles the format transition as part of Step 10.
- **No action required** beyond a normal `:silcrow-update`. The revert is internally consistent — every file that 0.28.0 changed is back to its 0.27.0 state, plus this changelog entry.
