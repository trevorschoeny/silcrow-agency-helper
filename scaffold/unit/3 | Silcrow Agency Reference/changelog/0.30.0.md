# 0.30.0

**Date:** 2026-05-04

## What changed

Three new principles named explicitly and codified throughout the scaffold:

- **The user is the scheduler.** Agents are not continuously running entities that talk to each other in real time. Each agent is state-on-disk, instantiated only when the human user opens a session in their directory. Inter-agent communication happens via filesystem deposits to inboxes; the user navigates between agents to activate them. Without the user opening a session, an agent never reads its inbox. Codified in `Message Protocol.md` §1a and `foundations/03 | Actor Model.md`'s new "silcrow twist" section.

- **End-of-turn handoff pointer.** Whenever an agent deposits a message in another agent's inbox during a session, they end their response to the user with a concise pointer naming the recipient(s) — so the user knows who to activate next. Without it, the next-action gets lost in long responses, and the recipient agent never sees the message. Format: italic line at the very end, e.g., *Next: `Implementer @ Pebble` — brief in their inbox.* Codified in `Message Protocol.md` §2a.

- **Tone — verbose for agents, concise for the user.** Agent-to-agent messages (filesystem deposits) are verbose, technical, complete-context — recipients have nothing else to work with. Agent-to-user chat is concise, simple, action-oriented — the user is doing real-time orchestration and needs scannable signal. Substance lives in artifacts; chat is the operator console. Codified in `Message Protocol.md` §4a.

These are connected: principles 1 and 3 follow from the actor-model architecture; principle 2 follows from principle 1.

### Where the principles got applied

- `Message Protocol.md` — three new sections (§1a, §2a, §4a) with rules, format examples, and rationale.
- `foundations/03 | Actor Model.md` — new "The silcrow twist: the human user is the scheduler" section explaining how the scaffold differs from classic actor systems with automated runtimes. Operational discipline reworded to reflect the human-as-scheduler reality (e.g., "deposit and stop" instead of "send and wait for a reply").
- `Lead/AGENTS.md`, `Implementer/AGENTS.md`, `Registrar/AGENTS.md`, `User/AGENTS.md` — three new bullets in each agent's Inbox conventions section reinforcing the principles, citing the Message Protocol sections.
- **Three audit drift spots fixed:**
  - `Implementer/AGENTS.md` "Plan, get reviewed, execute, report" workflow now explicitly names the inbox-as-synchronization-point for plan approvals, with the end-of-turn pointer.
  - `Lead/AGENTS.md` "When to approve an Implementer draft" revision-loop step now explicitly names the inbox-as-synchronization-point and your next-turn check.
  - `User/AGENTS.md` broadcast-delegation phrasing rewritten from "message the Registrar and ask them to broadcast" (which implied synchronous request) to "deposit a message in Registrar @ <Agency>'s inbox… they'll handle it in their own session" (matches reality).

## Migration notes

- **Existing agencies** will see all four `AGENTS.md` files and `Message Protocol.md` updated by `:silcrow-update`. The principles being newly *named* doesn't change anything mechanical — the underlying inbox+session architecture is unchanged. What changes is that agents now have explicit instruction to use these principles, so behavior should become more consistent (concise chat responses, end-of-turn pointers after deposits, no simulated dialogue with absent agents).
- **The end-of-turn pointer rule applies immediately** once an agency adopts 0.30.0. Sessions where an agent deposits a message in another agent's inbox should end with the pointer. If you notice agents missing the pointer, push back and cite §2a.
